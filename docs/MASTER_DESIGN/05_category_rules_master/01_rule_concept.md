## 1. 分類ルールの目的

### 1.1 分類ルールとは

**分類ルール**（`category_rules`）は、CSV インポート時に**店名からカテゴリを自動判定するためのルール集**です。

**主な目的**:

- 🤖 **自動分類**: CSV 取引データの店名から自動的にカテゴリを判定
- 📊 **分類精度向上**: ユーザーがルールを追加・編集することで分類精度を改善
- 🎯 **カスタマイズ**: ユーザーの利用店舗に合わせた分類ルールの管理

### 1.2 分類ルールの役割

```
CSV インポート時
   ↓
店名: "セブンイレブン 渋谷店"
   ↓
分類ルール照合
   ↓
ルール: keyword="セブン" → category_id=2 (食費)
   ↓
取引データに category_id=2 を設定
```

### 1.3 使用箇所

- **CSV 自動分類**: 取引データの `store_name` と照合
- 🏷️ **CategoryRulesPage**: ルール一覧・追加・編集・削除

## 2. 設計方針

### 2.1 基本設計原則

| 原則               | 説明                                               |
| ------------------ | -------------------------------------------------- |
| **キーワード単位** | 1 つのルール = 1 つのキーワード                    |
| **優先度制御**     | 複数ルールがマッチする場合、優先度が高いものを採用 |
| **ユーザー管理**   | ユーザーが追加・編集・削除可能                     |
| **正規化処理**     | 半角カタカナ・大小文字・記号を統一してマッチング   |

### 2.2 設計の特徴

#### 2.2.1 シンプルな構造

- **1 ルール = 1 キーワード**: 複雑な正規表現ではなく、シンプルなキーワードマッチング
- **部分一致**: キーワードが店名に含まれればマッチ
- **優先度順**: 複数マッチ時は優先度が高いものを採用

#### 2.2.2 拡張性

- **ユーザー追加可能**: ユーザーが独自のルールを追加できる
- **優先度調整**: 既存ルールの優先度を調整可能
- **カテゴリ変更**: ルールのカテゴリを変更可能

## 3. キーワード単位での管理

### 3.1 キーワードの定義

**キーワード**は、店名に含まれる文字列です。

**例**:

- `セブン` → セブンイレブンにマッチ
- `JR` → JR 運賃にマッチ
- `マック` → マクドナルドにマッチ

### 3.2 キーワードの選定基準

#### 3.2.1 選定基準

| 基準               | 説明                           | 例                     |
| ------------------ | ------------------------------ | ---------------------- |
| **店舗名の一部**   | 店舗名に含まれる特徴的な文字列 | `セブン`, `ローソン`   |
| **略称**           | 店舗名の一般的な略称           | `マック`, `スタバ`     |
| **ブランド名**     | ブランド名やチェーン名         | `JR`, `ENEOS`          |
| **汎用キーワード** | 複数の店舗に適用可能           | `スーパー`, `コンビニ` |

#### 3.2.2 キーワードの長さ

- **推奨**: 2 ～ 10 文字程度
- **短すぎる場合**: 誤マッチの可能性（例: `カ` は汎用的すぎる）
- **長すぎる場合**: マッチしにくくなる（例: `セブンイレブン渋谷店` は具体的すぎる）

### 3.3 キーワードの重複

#### 3.3.1 システム制約

- **keyword の UNIQUE 制約**: 同じキーワードは 1 つのルールにしか登録できない
- **カテゴリとの組み合わせ**: 同じキーワードでも異なるカテゴリに登録可能（将来的な拡張）

#### 3.3.2 実務上の考慮

- **優先度で制御**: 同じキーワードが複数のカテゴリにマッチする場合、優先度で制御
- **具体性の高いルール**: より具体的なキーワードに高い優先度を設定

## 4. 優先度による制御

### 4.1 優先度とは

**優先度**（`priority`）は、複数のルールがマッチする場合に、どのルールを採用するかを決定するための数値です。

**特徴**:

- **高いほど優先**: 優先度が高いルールが優先される
- **範囲**: 0 ～ 999（将来的に拡張可能）
- **デフォルト**: 50

### 4.2 優先度の設定基準

#### 4.2.1 優先度の目安

| 優先度範囲   | 信頼度 | 使用例                                       |
| ------------ | ------ | -------------------------------------------- |
| **100 以上** | 高     | 特定の店舗名（`セブン`, `JR`）               |
| **50 ～ 99** | 中     | 一般的な店舗タイプ（`スーパー`, `コンビニ`） |
| **1 ～ 49**  | 低     | 汎用的なキーワード（将来の拡張用）           |

#### 4.2.2 優先度設定例

**食費カテゴリ**:

- `セブン` (priority: 105) - 特定の店舗名
- `マック` (priority: 104) - 特定の店舗名
- `スーパー` (priority: 100) - 汎用的なキーワード

**マッチング例**:

```
店名: "セブンイレブン 渋谷店"
  ↓
マッチするルール:
  - セブン (priority: 105) ✅ 採用
  - スーパー (priority: 100) - マッチするが優先度が低い
```

### 4.3 優先度による制御フロー

```
1. 店名を正規化
   ↓
2. 全ルールを優先度順（降順）で照合
   ↓
3. 最初にマッチしたルールを採用
   ↓
4. 該当カテゴリを返す
```

**実装例**:

```ruby
# 優先度順で照合
CategoryRule.by_priority.find_each do |rule|
  if normalized_store_name.include?(normalized_keyword)
    return rule.category  # 最初にマッチしたルールを返す
  end
end
```

## 5. まとめ

### 5.1 分類ルールの重要性

分類ルールは、以下の目的で重要です：

- **自動分類の精度**: 分類ルールの品質が自動分類の精度に直結
- **ユーザーカスタマイズ**: ユーザーが独自のルールを追加できる
- **継続的改善**: 分類精度を継続的に向上させることができる

### 5.2 設計のポイント

- **シンプルな構造**: キーワード単位での管理により理解しやすい
- **優先度制御**: 複数マッチ時の制御が可能
- **拡張性**: ユーザーが追加・編集・削除可能
