## 2025年8月10日 - プロジェクト始動

### 企画・設計フェーズ完了

今日は家計簿アプリプロジェクトの企画・設計フェーズを一気に進めた。正直、こんなに包括的な設計書を書くのは初めてで、途中で「本当にこれ全部作れるのか？」という不安が何度も頭をよぎった。

#### ドキュメント整備
- 要件定義書、機能仕様書、システム設計書を分けて作成
- 最初は全部混在していて、「あれ、これって仕様？要件？設計？」って混乱した
- でも分けてみると、それぞれの役割がはっきりして、むしろ頭の中が整理された

#### 技術選択での検討事項

**Vue.js vs React**
- Reactの方が求人多いし、情報も豊富
- でもVue.jsの方がとっつきやすそう
- 最終的にVue.js選択 → 学習コスト重視

**Rails vs Express vs FastAPI**
- Rails: 慣れてるけど重い？
- Express: 軽いけどBoilerplateが多い
- FastAPI: モダンだけど未経験
- 結局Railsに → 慣れてる技術で確実に進めたい

### 設計で工夫したポイント

#### カテゴリ自動分類の仕組み
最初は「AIで自動分類できたらかっこいいな」と思ったけど、現実的に考えると：

1. **キーワードマッチング**: 「セブンイレブン」→「食費」みたいなシンプルなルール
2. **正規表現**: もう少し柔軟なパターンマッチング  
3. **ユーザー学習**: 手動修正を覚えて精度向上

この3段階で進めることにした。最初からAIは無謀だけど、将来的には機械学習も検討したい。

#### フェーズ分けの重要性
「全部一度に作ろう」と思って挫折するパターンを避けたかった：

- **Phase 1**: フロントエンドのみ（ブラウザ完結）
- **Phase 2**: バックエンド追加  
- **Phase 3**: 認証・マルチユーザー

### データ構造設計で学んだこと
楽天カードのCSV形式を実際に見ながら設計したけど、「支払月」とか「繰越残高」とか、クレジットカード特有の項目が多い。将来他社対応するときは、この辺りの差異をうまく吸収する必要がありそう。

#### バックエンド設計・実装開始 🚧
**Rails 7 API モード** での開発に着手：

**重要Gem選定**
```ruby
gem "rack-cors"                  # フロントエンド連携用
gem 'csv'                        # 楽天カードCSV解析用  
gem 'groupdate'                  # 日付集計用
```

**データベース設計実装**
システム設計書通りに以下を実装：

```ruby
# Categories テーブル設計のポイント
t.string :name, null: false           # 一意制約でデータ整合性
t.string :color, null: false          # HEXカラーでフロントエンド連携
t.integer :display_order, default: 0 # ソート順制御

# Transactions テーブル設計のポイント
t.references :category, foreign_key: true # null許可（未分類対応）
t.decimal :amount, precision: 12, scale: 2 # 金銭計算の精度確保
t.text :raw_data                           # CSV生データ保存（トレーサビリティ）
```

### CSV解析の実装戦略
楽天カードCSVの実データを分析：
```csv
"利用日","利用店名・商品名","利用者","支払方法","利用金額",...
"2024/06/30","ｼﾞﾔﾊﾟﾝﾐ-ﾄｵﾛｼｳﾘｲﾁ","本人","1回払い","171",...
```

**課題発見：**
- 半角カタカナの店名 → 正規化が必要
- 日付形式が YYYY/MM/DD → Date.parse で対応
- 金額に円マークなし → to_f で直接変換可能

## 2025年8月13日 - プロジェクト完成

### 🎯 プロジェクト概要
楽天カードのCSVファイルをインポートして家計簿を管理できるWebアプリケーションを開発。モックアップを基に、フルスタックでの実装を完了。

### 📝 実装内容

#### 1. フロントエンド開発
- **技術スタック**: Vue 3 + Vite + Chart.js + Pug + SCSS
- **主要機能**:
  - CSVファイルアップロード機能
  - 月毎支出グラフ（Chart.js使用）
  - カテゴリ別支出割合円グラフ
  - 日別支出推移線グラフ
  - 月選択ナビゲーション（矢印＋クリック選択）
  - レスポンシブデザイン

#### 2. バックエンドAPI開発
- **技術スタック**: Rails 7 API + SQLite + groupdate gem + kaminari gem
- **主要機能**:
  - CSV一括インポートAPI
  - カテゴリ自動分類システム
  - 月次集計API
  - 分析データAPI
  - 取引データCRUD API
  - CORS設定完了

#### 3. データベース設計
```sql
-- カテゴリテーブル
categories: id, name, color, icon, description, display_order

-- 取引データテーブル  
transactions: id, category_id, transaction_date, store_name, amount, 
              payment_method, user_name, payment_month, raw_data, auto_classified
```

#### 4. CSVインポート機能
- **対応フォーマット**: 楽天e-NAVIのCSV形式
- **エンコーディング対応**: BOM付きUTF-8、Shift_JIS自動変換
- **文字種対応**: 半角カタカナ→全角カタカナ変換
- **エラーハンドリング**: 不正データのスキップと詳細ログ
- **実績**: 86/87件の正常インポート達成

#### 5. カテゴリ自動分類
店舗名パターンマッチングによる自動分類：
- 食費: セブン-イレブン、マクドナルド、マツヤ等
- 交通費: トウダイノウガク、メトロ等
- 日用品費: ダイソー等  
- その他カテゴリも網羅

### 🚀 技術的成果

#### 問題解決事例
1. **CSVパースエラー対応**
   - エラー: `Illegal quoting in line 1`
   - 解決: `liberal_parsing: true`オプション追加、BOM除去処理実装

2. **エンコーディング問題**
   - エラー: `"\xEF" from ASCII-8BIT to UTF-8`
   - 解決: バイナリ読み取り→BOM検出→適切なエンコーディング変換

3. **CORS設定**
   - 問題: フロントエンドからのAPI呼び出しでアクセス拒否
   - 解決: `cors.rb`でlocalhost:3002を許可

4. **Chart.js連携**
   - 問題: サンプルデータが表示されたまま
   - 解決: 実際のAPIデータを使用するよう全面書き換え
