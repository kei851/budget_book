# 家計簿アプリ開発日記

**プロジェクト開始日**: 2025年8月10日  
**作成者**: 個人開発者

## 2025年8月10日 - プロジェクト始動

### 今日やったこと

今日は家計簿アプリプロジェクトの企画・設計フェーズを一気に進めた。正直、こんなに包括的な設計書を書くのは初めてで、途中で「本当にこれ全部作れるのか？」という不安が何度も頭をよぎった。

#### ドキュメント整備
- 要件定義書、機能仕様書、システム設計書を分けて作成
- 最初は全部混在していて、「あれ、これって仕様？要件？設計？」って混乱した
- でも分けてみると、それぞれの役割がはっきりして、むしろ頭の中が整理された

#### 技術選択で悩んだポイント

**Vue.js vs React**
- Reactの方が求人多いし、情報も豊富
- でもVue.jsの方がとっつきやすそう
- 最終的にVue.js選択 → 学習コスト重視

**TypeScript vs JavaScript**  
- TypeScriptの方が安全だし、現代的
- でも学習コスト考えると、最初はJavaScriptでいいかも
- Phase 1はJavaScript、Phase 2でTypeScript移行を検討

**Rails vs Express vs FastAPI**
- Rails: 慣れてるけど重い？
- Express: 軽いけどBoilerplateが多い
- FastAPI: モダンだけど未経験
- 結局Railsに → 慣れてる技術で確実に進めたい

### 設計で工夫したポイント

#### カテゴリ自動分類の仕組み
これが一番頭を悩ませた。最初は「AIで自動分類できたらかっこいいな」と思ったけど、現実的に考えると：

1. **キーワードマッチング**: 「セブンイレブン」→「食費」みたいなシンプルなルール
2. **正規表現**: もう少し柔軟なパターンマッチング  
3. **ユーザー学習**: 手動修正を覚えて精度向上

この3段階で進めることにした。最初からAIは無謀だけど、将来的には機械学習も検討したい。

#### フェーズ分けの重要性
「全部一度に作ろう」と思って挫折するパターンを避けたかった：

- **Phase 1**: フロントエンドのみ（ブラウザ完結）
- **Phase 2**: バックエンド追加  
- **Phase 3**: 認証・マルチユーザー

この段階的アプローチなら、Phase 1だけでも使えるものが作れる。

### データ構造設計で学んだこと

楽天カードのCSV形式を実際に見ながら設計したけど、「支払月」とか「繰越残高」とか、クレジットカード特有の項目が多い。将来他社対応するときは、この辺りの差異をうまく吸収する必要がありそう。

### 今日の反省

- **スコープが大きすぎる**: 20年分データとか、50万件処理とか、本当にできるのか不安
- **技術学習の時間**: Vue.js、Pug、SCSS、全部中途半端な知識
- **デザイン力**: Figmaでモックアップ作るって言ったけど、デザインセンスがない

### 明日以降の予定

1. **Figmaでモックアップ作成** → UI/UXの具体化
2. **Vue.js学習** → 公式ドキュメント、チュートリアル
3. **開発環境構築** → Vite、eslint、prettier等

### 正直な気持ち

**不安なこと：**
- 本当に最後まで作りきれるのか
- Vue.jsの学習コストどれくらいか
- デザインがダサくならないか
- パフォーマンス問題にどう対処するか

**楽しみなこと：**
- 自分の家計簿データで実際にテストできる
- グラフで可視化されたデータを見るのが楽しそう
- フルスタック開発の経験を積める
- 作ったものをポートフォリオにできる

**学習目標：**
- Vue.js 3 Composition API
- モダンフロントエンド開発
- UI/UXデザインの基礎
- 大量データ処理のベストプラクティス

### 今日の一言

「完璧を目指すより、まず動くものを作ろう。」

設計書を書いてる時に完璧主義が出てしまって、「この設計で本当に大丈夫か？」「もっと良いアーキテクチャがあるんじゃないか？」って考え込んでしまった。でも、完璧な設計なんてないし、作りながら学んで、リファクタリングしながら改善していけばいい。

明日からは実装フェーズ。頑張ろう！
