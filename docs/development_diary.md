## 2025年8月10日 - プロジェクト始動

### 企画・設計フェーズ完了

今日は家計簿アプリプロジェクトの企画・設計フェーズを一気に進めた。正直、こんなに包括的な設計書を書くのは初めてで、途中で「本当にこれ全部作れるのか？」という不安が何度も頭をよぎった。

#### ドキュメント整備
- 要件定義書、機能仕様書、システム設計書を分けて作成
- 最初は全部混在していて、「あれ、これって仕様？要件？設計？」って混乱した
- でも分けてみると、それぞれの役割がはっきりして、むしろ頭の中が整理された

#### 技術選択での検討事項

**Vue.js vs React**
- Reactの方が求人多いし、情報も豊富
- でもVue.jsの方がとっつきやすそう
- 最終的にVue.js選択 → 学習コスト重視

**Rails vs Express vs FastAPI**
- Rails: 慣れてるけど重い？
- Express: 軽いけどBoilerplateが多い
- FastAPI: モダンだけど未経験
- 結局Railsに → 慣れてる技術で確実に進めたい

### 設計で工夫したポイント

#### カテゴリ自動分類の仕組み
最初は「AIで自動分類できたらかっこいいな」と思ったけど、現実的に考えると：

1. **キーワードマッチング**: 「セブンイレブン」→「食費」みたいなシンプルなルール
2. **正規表現**: もう少し柔軟なパターンマッチング  
3. **ユーザー学習**: 手動修正を覚えて精度向上

この3段階で進めることにした。最初からAIは無謀だけど、将来的には機械学習も検討したい。

#### フェーズ分けの重要性
「全部一度に作ろう」と思って挫折するパターンを避けたかった：

- **Phase 1**: フロントエンドのみ（ブラウザ完結）
- **Phase 2**: バックエンド追加  
- **Phase 3**: 認証・マルチユーザー

### データ構造設計で学んだこと
楽天カードのCSV形式を実際に見ながら設計したけど、「支払月」とか「繰越残高」とか、クレジットカード特有の項目が多い。将来他社対応するときは、この辺りの差異をうまく吸収する必要がありそう。

**データベース設計実装**
システム設計書通りに以下を実装：

```ruby
# Categories テーブル設計のポイント
t.string :name, null: false           # 一意制約でデータ整合性
t.string :color, null: false          # HEXカラーでフロントエンド連携
t.integer :display_order, default: 0 # ソート順制御

# Transactions テーブル設計のポイント
t.references :category, foreign_key: true # null許可（未分類対応）
t.decimal :amount, precision: 12, scale: 2 # 金銭計算の精度確保
t.text :raw_data                           # CSV生データ保存（トレーサビリティ）
```

### CSV解析の実装戦略
楽天カードCSVの実データを分析：
```csv
"利用日","利用店名・商品名","利用者","支払方法","利用金額",...
"2024/06/30","ｼﾞﾔﾊﾟﾝﾐ-ﾄｵﾛｼｳﾘｲﾁ","本人","1回払い","171",...
```

**課題発見：**
- 半角カタカナの店名 → 正規化が必要
- 日付形式が YYYY/MM/DD → Date.parse で対応
- 金額に円マークなし → to_f で直接変換可能

### 🚀 技術的成果

#### 問題解決事例
1. **CSVパースエラー対応**

2. **エンコーディング問題**
   - エラー: `"\xEF" from ASCII-8BIT to UTF-8`
   - 解決: バイナリ読み取り→BOM検出→適切なエンコーディング変換

3. **CORS設定**
   - 問題: フロントエンドからのAPI呼び出しでアクセス拒否
   - 解決: `cors.rb`でlocalhost:3002を許可

4. **Chart.js連携**
   - 問題: サンプルデータが表示されたまま
   - 解決: 実際のAPIデータを使用するよう全面書き換え

## 2025年8月25日 - チャートナビゲーション改善と色の統一化

### 🎯 今回の作業内容

前回の作業から継続して、主にユーザビリティとUI統一性の改善を行いました。

#### 1. チャートナビゲーション機能の完全修正

**問題**: 15ヶ月分のデータがあるにも関わらず、12ヶ月分しか表示されない状態が発生していた
- **原因**: HomePage.vue:133にて `.slice(-12)` でデータを人為的に制限していた
- **解決**: `Array.from(allMonths).sort()` に変更し、全期間データを表示するよう修正

**ナビゲーションボタンの活性制御**:
- 以前・以降ボタンの活性/非活性状態が正しく判定されていなかった
- ExpenseChart.vueとHomePage.vueの連携でナビゲーション状態を正確に同期

**初期表示の改善**:
- 最新の月が右端に表示されるよう `calculateInitialOffset` 関数を実装
- ユーザーが最も関心を持つ直近のデータが即座に見える状態に

#### 2. デバウンス機能の実装

**背景**: ユーザーが矢印ボタンを連打した際にタイムアウトエラーが発生する問題

**実装内容**:
- `isUpdating` フラグと500msのタイムアウトを設定
- Chart.js の無限ループエラーを防ぐため `animation: false` を追加
- ボタン連打による不整合状態を完全に回避

```javascript
const updateChart = async () => {
  if (isUpdating.value) return
  isUpdating.value = true
  try {
    // Chart update logic
  } finally {
    setTimeout(() => { isUpdating.value = false }, 500)
  }
}
```

#### 3. カテゴリ色の統一化

**目的**: プルダウンの色とグラフの色を対応させて、視覚的な一貫性を向上

**実装内容**:
- CategoryRulesPage.vue でハードコードされたカテゴリデータを実際のデータベースからの取得に変更
- プルダウンオプションに `:style` バインディングでカテゴリ色を適用
- カテゴリチップにも対応する色を適用し、全体的な色の統一性を実現

```vue
option(
  v-for="category in categories" 
  :key="category.id" 
  :value="category.id"
  :style="`background-color: ${category.color}; color: white;`"
) {{ category.name }}
```

### 🛠️ 技術的な課題と解決策

#### Chart.js の無限ループエラー
**問題**: "Cannot read properties of null (reading 'save')" エラーが頻発
**解決**: 
- アニメーション無効化による描画の安定性向上
- Canvas要素の存在確認を強化
- 適切な更新タイミングの制御

#### Vue.js の emit定義エラー
**問題**: `emit is not defined` エラー
**解決**: setup関数のパラメータに `{ emit }` を追加し、適切にemitを受け取る

#### Pug テンプレートでの日本語テキスト
**問題**: 日本語テキストがPugテンプレート内で正しく表示されない
**解決**: テキスト前に `|` (パイプ文字) を追加してプレーンテキストとして処理

### 緊急修正対応

前セッションの最後で残ったVueコンパイラエラーを修正しました。

#### 発生していた問題
- `AnalyticsPage.vue` で `loadAvailableMonths` 関数が重複宣言
- Vue compiler error: "Identifier 'loadAvailableMonths' has already been declared" 
- 282行目と381行目で同じ関数名が使用されていた

#### 解決アプローチ
1. **エラー特定**: 重複宣言箇所を Read ツールで確認
2. **機能比較**: 2つの実装のうち、より包括的な機能を残すべきか判断
3. **統合実装**: トランザクションデータベース実装を採用し、古い実装を削除

#### 技術的実装
- **採用した実装**: `apiService.getTransactions()` を使用して実際のトランザクションデータから利用可能月を抽出
- **削除した実装**: `apiService.getMonthlyData()` の `monthly_totals` を使用する古い方式
- **メリット**: データの整合性がより高い、実際のデータベース状態を正確に反映

#### 解決結果
- ✅ Vue.js開発サーバーが正常にコンパイル
